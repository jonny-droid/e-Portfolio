Unit 2: Information Systems and their Importance

Please see here for my reflections

Shelly et al (2016) suggested three core parts of an information system:
  • The Harware (e.g. computers, keyboard etc...)
  • The software (made up of the OS and Application software) - thse are essentially the instructions that tell the hardware what to do
  • Data
  • People (e.g. IT staff)
  • Processes

It has been suggested that communications should be added to the above. However, technically, Networking belongs in both Hardware and Software.

The above is what separates Information Systems from Computer Science (CS)). CS is one part of Information Systems.

Object-Oriented programming uses a modular approach, making easier to maintain and ensure flexibility, as blocks can be reused.

A class is like a form/questionnaire. Many instances can be created from a single class.

A class is a blueprint for how something should be achieved, but contains no data itself. In the example below the class of 'dog' specifies its name and age, but does not contain the name of age of any specific dogs.

Within relational datbases there are some analogies to OOP (Object-Oriented Programming):

  • States/variables are similar to attributes
  • Inheritence and composition are similar to relationships
There are four main concepts within OOP: encapsulation, inheritence, polymorphism and abstraction.

UML is an industry standard method for graphically showing the artifacts of a software system.
This is used to save time and costs when creating and modifying information systems.
It also also means it is possible to gain a high level overview of a system to see if it meets the specification.
UML can be static (shows structure e.g. class diagram) or dynamic (showing a behaviour, e.g. a sequence diagram).
Class diagrams are one of the most commonly used forms of UML diagram.

Tips from these two weeks:
  • A scenario is a sequence of steps desciribing hte interaction between a user and the system
  • Abstract class: only exists to be inherited (opposite of concrete class)
  • Actors are people or systems who interact with the subject
  • Functional requirements are what the system must and must not do (e.g. adds customers).
    Non-functional requirements are the product properties and user expectations (e.g. it's reliable)
  • It is common to leave out the attributes at the early stage of making the UML diagrams
  • There are lots of ways to make the requirements, and there is often not a right or wrong answer

A database is a structured approach to data storage.
A database = operational data + a data dictionary (describes strucutre, of database, including relationships between diferent elements, access and manipulation of data).

Three components make up a database:

  • Entities: take the form of a table
  • Attributes: the elements that go into making up an entity (they make up a column)
  • Relationships: how two entities are linked together. Allows us to store linked ata across several tables.

One of the core elements of relational databases are primary (unique identifier in a dable) and foreign keys (a colum in one table that references another table).
NB: a composite key is where two fields create a unique identifer. E.g. 1st line of an address and postcode form a primary key when combine

Normalisation aims to ensure the integrity of data and gets rid of redudnant data through stopping multiple copies of the same attribute=
Three rules of normalisation:
  • 1st normal form:
    No collumns have repeated or similar data. E.g. [First name, Surname, Full name]
    Each item cannot be broken down any further (it is atomic (cannot be deposed further, e.g. full name can be decmposed into first name and surname))
    Each row is uniqe, i.e. it has a primary key
    Each field has a unique name (e.g. you not: shoes, colour, colour)
  • 2nd normal form:
    Any table that adheres to the 1st normal form and has a simple primary key is automatically in the 2nd normal form. Everything in the table should relate to the primary key (if composite primary key then all data needs to relate to both parts of this). E.g. should not have [artist, venue, style] if venue and artist are a compound primary key, as style only depends on artist.
  • 3rd normal form: there are no non-key attributes that depend on anotehr non-key attribute. I.e. a value cannot be obtained by refering to another attribute. E.g. [artist, date, city, country], [city,country] should form its own table.

Big data and NoSQL

Big Data: a field looking at how we can handle the huge amount of data we generate as a society
IBM discussed fours Vs to distinguish Big Data from regular data (reference!!!):
  • Volume (amount of data)
  • Velocity (speed generated and the amount that it moves around)
  • Variety (unstructured and difficult to store in a table)
  • Veracity (accuracny and trustworthiness)
NoSQL (Not only SQL) is associated with Big Data. NoSQL is not a single technology, but an umbrella term for database technologies which are flexible, scalable and fast - and are particlarly appropriate for Big Data.
NoSQL is not limitedt to storing data in columns or having conformity across rows. Where database integrity is important, for example with payment systems, then may wish to rely on SQL. NoSQL is often part of a wider database management system, including both SQL and noSQL storage. There are four types of NoSQL databases (not all NoSQL database types fit neatly within these categories):
  • Key-Value: records being stored in pairs (one key to one value). A key is an arbitrary string, for example a file name. A value can be anything as short as a single number or as long as multiple lines of code (such as those stored in a .html file). The value can also be a list.
  • Document: each record stored in a document. For example, a medical patietn record documenting everything
  • Wide-column: columns can vary from row to row. Tables can be created, adjusted and dropped whilst runnin a query
  • Graph: uses nodes instead of tables.

The Model-View-Controller (MVC):
This is a way of thinking about software logic.
The Model represents the data (and nothing else), the View what data is displayed to the user, and the Controller acts between the Model and View (by interpreting user inputs).

Web frameworks:
A framework is a directly containing resources such as libraries, reference documentation, database access etc… that has already been developed to help you develop your application. Reusable code provides functionality within your programmes. It’s essentially so that you can avoid needing to reinvent the wheel. Every programming language has its own frameworks (for example JQuery and Javascript). A web framework includes a built in server and the delegation of responsibility for various parts of your application. It will automate implementation for various common solutions, such as HTTPS request handling. Flask is an example of Python web framework (Django is another common example). Flaks offers templating. A template mostly contains HTML, however when you want to add any dynamic behaviour to a webpage you use Jinja.

Malicious code injections:
In the case of an SQL injection, this sends untrusted code to the web server, e.g. through someone entering SQL code into the URL to fool the web server.
