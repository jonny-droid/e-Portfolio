My initial post

Cross site scripting

This post looks at OWASP A1:2017-Injection (OWASP, 2020). The activity diagram specifically demonstrates reflected Cross-Site Scripting (XSS). This is a client-side attack where a hacker constructs a custom URL which executes JavaScript in the victim's browser. This can be used to carry out an action or to send data to the hacker (Ambedkar and Ambedkar 2016).

Other examples of cross-site scripting includes (Dora and Nemoga, 2021):

  • Dom based XSS: attackers add malicious code to a webpage. This is a client-side attack. Dora and Nemoga (2021) state that some authors regard this as a type of reflected attack, however it does not include any data being sent to the server.
  • Stored/persistent XSS: attackers store malicious code on a website's database. This is often accomplished through a search field, with a hacker adding Javascript/HTML into a search box or form. This is a server-side attack.

Rodriguez et al (2020) list some methods to prevent cross-site scripting:

  • Blacklist certain user inputs, for example “<” and “>”
  • Escaping characters so code is interpreted as text and not run.
  • Using open sources XSS libraries.
  • Tools exist that can be used to automatically detect cross site scripting vulnerabilities (OWASP 2021). OWASP lists nine of these.

My Summary post

During the first unit we covered the Software Development Lifecycle (SDLC), the difference between architecture and design (including reference to TOGAF), predictive (waterfall) and adaptive (agile) software methodologies, and we looked at some of the relevant standardisation bodies. OWASP's top ten security vulnerabilities for web applications were explored. I wrote my initial post based on OWASP (2020) A1:2007-injection and A7:2007- Cross-Site Scripting XSS (focussing on cross-site scripting attacks (XSS)). I had some valuable feedback stating that one potential solution to XSS attacks (using open source XSS libraries) can also come with security threats. This could mean the packages themselves execute malicious script (there is often a lack of moderation within software language libraries) (Lakshmanan, 2021).

I also received useful feedback on my UML. Where I had a decision box stating “the website has a way to mitigate threats, for example escaping character”, I would be better to use a more definitive question such as “does the website use escaping character?”. The former statement leaves an open-ended number of potential methods of mitigation and lacks detail. This made me think that I could add another decision box asking if the web application blacklists certain characters such as “<” and “>", however I decided not to as my reading suggested that escaping characters is the most comprehensive defence against XSS attacks.

In unit 2 we covered the background of UML and the different types of diagrams that can be used at different points of the SDLC. During unit 3 we learned about the history of programming languages, and the difference between compiled and interpreted languages (and the use of hybrid code). My further reading helped me understand the paradigms within the categories of iterative and declarative programming. I also learned about static and dynamic typed languages, and the advantages and disadvantages of these in different situations.
