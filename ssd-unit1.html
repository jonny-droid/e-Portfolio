Cross site scripting

This post looks at OWASP A1:2017-Injection (OWASP, 2020). The activity diagram specifically demonstrates reflected Cross-Site Scripting (XSS). This is a client-side attack where a hacker constructs a custom URL which executes JavaScript in the victim’s browser. This can be used to carry out an action or to send data to the hacker (Ambedkar and Ambedkar 2016).

Other examples of cross-site scripting includes (Dora and Nemoga, 2021):

  • Dom based XSS: attackers add malicious code to a webpage. This is a client-side attack. Dora and Nemoga (2021) state that some authors regard this as a type of reflected attack, however it does not include any data being sent to the server.
  • Stored/persistent XSS: attackers store malicious code on a website’s database. This is often accomplished through a search field, with a hacker adding Javascript/HTML into a search box or form. This is a server-side attack.

Rodriguez et al (2020) list some methods to prevent cross-site scripting:

  • Blacklist certain user inputs, for example “<” and “>”
  • Escaping characters so code is interpreted as text and not run.
  • Using open sources XSS libraries.
  • Tools exist that can be used to automatically detect cross site scripting vulnerabilities (OWAST 2021). OWAST lists nine of these.

Feedback 2

Hi Jonathan. It seems that the issue of updating or patching vulnerability of an IT system, including the outdated Microsoft Window of your case, is a rather common cause of the information system failure. Similar failure occurs in the case I searched. Data of the Equifax breached through a vulnerable Apache Struts Web Framework server, which was not patched since the administrator missed the critical security patch. In addition to an out of date mailing list, the system administrator also failed to receive the email sent from the security officer regarding the incorrectly configured database.

Therefore, human or administration failure serves as one of the main causes in both incidents. Higher priority should be given to continuous updating of the operation system and all software related to the IT system and security vulnerability patching for any information system, which are suggested to be conducted on a regular and restricted basis. For instance, as suggested by the Secure Team (2019), monthly patching cycles can be better implemented efficiently by using network segmentation and automation.

Fruhlinger, J. (2020). Equifax data breach FAQ: What happened, who was affected, what was the impact?. Available from: https://www.csoonline.com/article/3444488/equifax-data-breach-faq-what-happened-who-was-affected-what-was-the-impact.html

SecureTeam (2019) 'Security Patching - The Stuff of Sys Admin Nightmares'. Available at: https://secureteam.co.uk/articles/infrastructure/security-patching-the-stuff-of-sys-admin-nightmares/
