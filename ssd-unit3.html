What is a secure programming language?

1) What factors determine whether a programming language is secure or not?

A perfect language would contain no buffer errors, no injection errors and no information leakage. No mainstream language can guarantee this. Some languages however may promote better security practices than others and minimise the ability for developers to write insecure code.

Where external packages/modules are used there is a certain amount of trust involved. However, there are libraries that exist to help check these.

2) Could Python be classed as a secure language? Justify your answer.

No in the sense that no language can be classed as secure.

Console vulnerabilities

Python 2 had certain vulnerabilities that Python 3 fixed. Python 3 converts inputs to strings. However, Python 2 does an eval() meaning that it executes a command for the input. This can be gotten around by using raw_input.

Buffer overflow attack

Buffer overflow – where data exceeds the amount of allocated memory to a specific point is caused when the application fails to check for the length of the input. Boundary checking on variable-length input helps to mitigate this. C and C++ are particularly vulnerable to this as their functions do not effectively check for boundaries.

A buffer is a temporary data storage area that has a limited space allocated for any single task. Programming should be written with error checking to make sure that the buffer receives the right type and right amount of data.

A buffer overflow occurs when an input given to an application exceeds the allocated buffer memory. The input is accepted and overwrites other data, crashes the program or can be used for data leakage.

Python3 is better than Python2 for manging memory and returning errors.

 

Server-side vulnerabilities

SSTI (server-side template injection). This is possible in Python web frameworks.

DDOS attacks. Can happen with Python web frameworks where arithmetical expressions are passed to the server causing it to crash.

SQL injection (is possible when Python is connected to MySQL through a connector). Need to use strategies to make sure output is escaped.

Most security issues are due to precautions not being taken. The developer needs to make sure that file descriptions are closed, passwords do not compare original data in memory but use strong password hashing libraries with in-built SALT and a fixed number of hashing rounds.

The developer needs to make sure that sensitive data is encrypted, and that the system up to date
